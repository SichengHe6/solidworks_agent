# 1. 这个系统为什么算 Agent，而不是普通工作流？

**面试官想看：**
你能不能区分 Agent 和 workflow。

**回答话术：**

这个系统有 workflow 的骨架，但它不是纯静态 workflow。它具备几个 Agent 特征：

```text
1. 能根据输入动态决定任务路径：单零件 / 装配体
2. 能调用工具：读写文件、执行代码、运行外部程序、获取日志和截图
3. 能基于执行反馈调整下一步动作
4. 有短期记忆和历史尝试，用于连续修复
5. 可以按需加载 skills，改变当前建模策略
```

所以它是：

> **Workflow-guided Agent**，不是完全自由游走的 autonomous agent。
> 它用固定流程保证稳定性，用 Agent 决策和工具调用保证灵活性。

更高级说法：

> 我不会把它包装成完全自主智能体，而是一个“有边界的 Agentic Workflow”。在工程应用里，这种受控 Agent 通常比完全开放式 Agent 更可靠。

---

# 2. 多 Agent 是否真的必要？为什么不用一个强模型端到端？

**面试官想看：**
你是否理解多 Agent 的 trade-off。

**回答话术：**

多 Agent 不是为了堆概念，而是因为任务天然有不同阶段：

```text
需求理解
任务规划
代码生成
结果校验
失败修复
```

这些阶段的目标函数不同：

```text
需求 Agent：追求信息完整
规划 Agent：追求结构清晰
建模 Agent：追求代码可执行
校验 Agent：追求发现问题
修复过程：追求最小变更
```

如果用一个 Agent 端到端做，容易出现：

```text
上下文太长
职责混杂
输出格式不稳定
失败后难以定位问题
校验和生成相互妥协
```

但我也会承认：

> 多 Agent 会增加通信成本、状态管理复杂度和错误传播风险。

所以更准确的设计原则是：

> 不是 Agent 越多越好，而是当不同阶段有明显不同目标、输出格式和校验标准时，才拆成多个 Agent。

---

# 3. 多 Agent 之间会不会出现错误传播？怎么控制？

**回答话术：**

会，这是多 Agent 系统最大的风险之一。

比如需求理解阶段漏掉尺寸，后面的规划和代码都会继承这个错误。

控制方式有几类：

```text
1. 结构化中间表示
   每个阶段输出 JSON，而不是自由文本。

2. 阶段间评审
   在需求后、规划后做 soft review。

3. schema 校验
   必须检查字段完整性、类型、引用关系。

4. 可恢复状态
   如果后面失败，可以回退到规划或建模阶段。

5. 保留不确定性
   对缺失信息写 assumptions，而不是假装确定。
```

可以补一句：

> Agent 间通信最怕“自然语言传话失真”，所以中间产物要尽量结构化、可验证、可回放。

# 4. skills 怎么选择？是关键词匹配还是模型判断？

**回答话术：**

可以分层做，不建议完全依赖模型自由选择。

一个比较稳的方案是：

```text
第一层：规则召回
根据零件类型、关键词、特征召回候选 skills。
例如：法兰、孔阵列、旋转体、倒角。

第二层：向量召回
用 embedding 匹配任务描述和 skill 描述。

第三层：LLM rerank
让模型根据 part_spec、约束和失败日志选择最相关的 skill。

第四层：执行反馈更新
如果某个 skill 对这类任务经常失败，降低优先级。
```

面试包装：

> skill selection 本质上是一个 retrieval + routing 问题，不应该只靠 prompt。更好的方式是规则、语义检索和模型 rerank 结合。

---

# 5. skills 加载后怎么避免上下文爆炸？

**回答话术：**

skills 不能全部塞进上下文，否则会出现 token 浪费和干扰。

解决方式：

```text
1. skill metadata 先检索
   先只看 skill 名称、适用场景、输入输出。

2. 分块加载
   只加载当前步骤需要的片段，比如 API 示例或错误修复片段。

3. 压缩模板
   把长文档变成 recipe：适用条件、步骤、关键 API、注意事项。

4. 失败后增量加载
   第一次只加载建模模板，失败后再加载 debug skill。

5. 上下文优先级
   用户需求 > 当前错误日志 > 当前 skill 片段 > 历史背景。
```

一句话：

> skills 要做按需加载和分层摘要，而不是全量注入。

# 6. skills 怎么评估好不好？

**回答话术：**

不能只看模型回答是否像样，要看执行指标。

可以从几个维度评估：

```text
1. 任务成功率
   加载 skill 后模型生成代码的执行成功率。

2. 首次成功率
   first-pass success rate。

3. 平均重试次数
   skill 是否减少 retry。

4. 错误类型分布
   是否减少 API 错误、路径错误、几何错误。

5. 泛化能力
   对同类但不同参数任务是否有效。

6. 负迁移率
   加载错误 skill 是否让结果变差。

7. token 成本
   skill 带来的上下文成本是否值得。
```

更高级一句：

> skill 的评价应该是 offline benchmark + online telemetry，而不是主观判断。

---

# 7. 失败重试会不会陷入死循环？

**回答话术：**

会，所以必须设计停止条件和升级策略。

可以设置：

```text
1. 最大重试次数
2. 错误类型分类
3. 连续相同错误检测
4. 修复差异检测
5. fallback 到更保守模板
6. 必要时向用户追问
```

比如连续三次都是同一个 API 报错，就不应该继续让模型自由修，而应该：

```text
切换 skill
使用保守模板
回退到规划阶段
或者请求用户补充约束
```

面试话术：

> 重试不是简单 while loop，而应该是 error-aware retry policy。

---

# 8. 怎么避免 Agent 越修越坏？

**回答话术：**

这个问题很关键。

解决办法：

```text
1. 保留上一次成功基线
   如果修复失败，可以回滚。

2. 使用局部编辑策略
   不让模型每次完全重写。

3. diff-based review
   检查修改是否只影响错误相关部分。

4. regression check
   之前已经通过的约束不能被破坏。

5. 历史尝试对比
   如果新版本比旧版本错误更多，丢弃新版本。
```

包装版：

> 修复 Agent 的目标不是“重新生成一个答案”，而是“在不破坏已满足约束的前提下最小化修复”。

---

# 9. 记忆怎么防止污染？

**面试官想看：**
你有没有意识到 memory 不是越多越好。

**回答话术：**

记忆污染主要来自：

```text
错误假设被长期保存
失败代码被当成正确经验
过期任务上下文影响新任务
用户临时修改被误认为长期偏好
```

解决方式：

```text
1. 记忆分层
   对话记忆、任务记忆、历史尝试、长期知识分开。

2. 记忆带状态
   标记 confirmed / assumption / failed / deprecated。

3. 作用域隔离
   每个任务 workspace 独立，避免跨任务污染。

4. TTL 或窗口限制
   短期记忆只保留最近几条。

5. 写入前校验
   只有通过验证的经验才进入长期 knowledge / skill。
```

一句话：

> Memory 要有 schema、scope 和 lifecycle，不能把所有文本都当知识存起来。

# 10. 这个系统的 planner 是真的推理，还是模板拼接？

**回答话术：**

可以这样回答得比较稳：

> 当前更像是 schema-constrained planning，不是完全开放式规划。

也就是说，它不是任意生成一个计划，而是在固定 schema 下生成：

```text
任务类型
零件列表
实例列表
接口
约束
执行顺序
输出路径
```

好处是：

```text
稳定
可校验
可被下游代码生成使用
```

不足是：

```text
灵活性不如自由规划
schema 设计决定上限
复杂任务可能需要分层规划
```

更高级说法：

> 工程 Agent 里的 planner 往往不是追求“完全自由思考”，而是追求“结构化、可执行、可验证”。

# 11. 你怎么保证 Agent 输出 JSON 是合法的？

**回答话术：**

有几种办法，从弱到强：

```text
1. Prompt 约束
   明确要求只输出 JSON。

2. JSON 解析失败重试
   把错误反馈给模型修复。

3. Schema validation
   检查字段、类型、引用关系。

4. Constrained decoding
   如果模型平台支持，直接用 JSON schema / grammar。

5. Function calling
   让模型输出结构化 tool call，而不是自由文本。
```

可以补一句：

> 工程上我更倾向用 function calling 或 schema-constrained output，而不是只依赖 prompt。

# 12. 工具调用怎么防 Prompt Injection？

**回答话术：**

风险是：用户或知识库内容可能诱导 Agent 调用危险工具，比如读写不该访问的路径。

防护方式：

```text
1. 工具白名单
   只能调用预定义工具。

2. 参数校验
   path 必须在 workspace 范围内。

3. 权限最小化
   Agent 不能直接执行任意系统命令。

4. 工具结果隔离
   检索到的知识只作为参考，不能覆盖系统策略。

5. 高风险操作审批
   删除文件、网络请求、外部命令需要额外确认。
```

一句话：

> Agent 的安全边界不能靠模型自觉，必须靠工具层强约束。

---

# 13. 如果工具执行结果和模型判断冲突，信谁？

**回答话术：**

优先信工具执行结果。

因为工具结果来自真实环境，比如代码是否运行成功、文件是否存在、日志是否报错。

模型判断只作为解释和修复建议。

可以说：

```text
工具结果是 ground truth
模型判断是 reasoning layer
```

如果模型说成功，但工具日志失败，系统必须按失败处理。

# 14. 怎么做 Agent 评估？

**回答话术：**

可以分层评估：

```text
需求理解层：
- 结构化字段准确率
- 缺失信息识别率
- 澄清问题质量

规划层：
- plan schema 合法率
- part / instance 拆分正确率
- 约束完整率

代码生成层：
- 语法通过率
- 执行成功率
- 首次成功率
- 平均重试次数

工具执行层：
- 模型文件生成率
- 日志错误率
- 截图生成率

闭环层：
- 最终任务成功率
- 平均耗时
- 平均 token 成本
- 失败恢复率
```

更高级：

> Agent 评估不能只看最终答案质量，要看每个中间状态是否可控、可恢复、可解释。

# 15. 如果用户需求不完整，Agent 应该默认假设还是继续追问？

**回答话术：**

要看缺失信息的影响程度。

```text
低风险缺失：
可以用工程默认值，并写入 assumptions。
例如默认材料、默认倒角半径。

高风险缺失：
必须追问。
例如关键尺寸、装配接口、运动自由度、保存目标。
```

包装版：

> 不是所有缺失都要追问，否则交互成本太高；也不是所有缺失都能假设，否则会偏离用户意图。需要按风险分级处理。

# 16. 你如何设计 Agent 的状态？

**回答话术：**

状态应该围绕任务生命周期设计，而不是围绕代码函数设计。

比如：

```text
需求澄清中
规划生成中
代码生成中
工具执行中
结果校验中
可恢复失败
不可恢复失败
完成
```

状态里应该包含：

```text
当前输入
当前产物
失败原因
可回退位置
最大重试次数
下一步动作
```

一句话：

> 状态的价值是让 Agent 流程可追踪、可恢复、可观测，而不是只做日志展示。

# 17. 怎么处理并发和资源竞争？

**回答话术：**

Agent 生成可以并发，但工具执行要看资源特性。

比如：

```text
多个零件的 planning / code generation 可以并发
但 SolidWorks 这种有状态外部工具要串行执行
```

所以合理策略是：

```text
上层并行：提高吞吐
底层锁控：保证稳定
```

如果资源更多，可以做：

```text
多 worker
任务队列
资源池
隔离环境
超时控制
```

# 18. 这个系统适合用 ReAct 吗？

**回答话术：**

部分适合，但不能完全照搬开放式 ReAct。

ReAct 的思想是：

```text
Reason → Act → Observe → Reason
```

这个系统确实符合：

```text
生成计划/代码 → 调工具执行 → 观察日志/截图 → 修复
```

但 CAD 任务需要更强约束，所以更适合：

> **Structured ReAct / Tool-augmented Workflow**

也就是：

```text
固定主流程
关键节点允许 Agent 调工具和反思
工具调用结果必须结构化记录
```

这样比开放式 ReAct 更安全稳定。

# 19. skills 和 tools 的区别是什么？

**回答话术：**

这是个很容易被问的问题。

```text
tools 是 Agent 能做的动作
skills 是 Agent 做事的方法和知识
```

举例：

```text
tool：运行 Python、读文件、截图
skill：如何建法兰盘、如何做孔阵列、如何修复保存失败
```

一句话：

> tools 连接外部世界，skills 提供解决问题的能力模板。

# 20. 这个项目如何支持持续学习？

**回答话术：**

可以把成功和失败的历史沉淀成 skills 或经验库，但不能直接无脑写入。

持续学习路径可以是：

```text
1. 收集任务日志
2. 标注失败类型和修复方式
3. 提取可复用修复 recipe
4. 人工或自动审核
5. 更新到 skill 库
6. 在后续任务中检索使用
```

强调一点：

> 线上执行记忆不能直接变成长知识，必须经过过滤和验证，否则会把错误经验固化。

# 21. 如果模型 hallucinate 一个不存在的工具怎么办？

**回答话术：**

工具层应该拒绝。

系统只能执行白名单工具。模型输出未知工具时：

```text
1. 返回 Unknown tool 错误
2. 把可用工具列表反馈给模型
3. 要求它重新选择合法工具
```

这说明：

> 工具可用性由系统决定，不由模型幻想决定。

# 22. 如果 LLM 输出的代码有安全风险怎么办？

**回答话术：**

要从执行环境和静态检查两边控制。

```text
1. 限制工作目录
2. 限制可访问文件路径
3. 禁止危险命令
4. 运行前做静态扫描
5. 使用隔离环境执行
6. 设置超时和资源限制
```

更高级说：

> LLM 生成代码必须默认不可信，只有通过 sandbox 和 policy check 后才能执行。

# 23. 如何避免 Agent 对截图的错误理解？

**回答话术：**

视觉校验不能单独依赖模型主观判断。

可以结合：

```text
1. 截图视觉模型判断
2. CAD 文件存在性检查
3. 关键尺寸/特征的程序化检查
4. 多视角截图
5. 和计划中的接口/特征做对齐检查
```

一句话：

> 视觉模型适合发现明显异常，但最终应该和结构化规则结合。

# 25. 如果面试官问“前沿性在哪里”怎么答？

**回答话术：**

这个项目的前沿性不在 CAD 本身，而在 Agent 应用架构上：

```text
1. Agentic Workflow
   用受控流程组织 LLM、工具和反馈。

2. Dynamic Skill Loading
   按任务动态加载 skills，而不是固定 prompt。

3. Tool-grounded Generation
   生成结果通过真实工具执行验证。

4. Multi-agent Specialization
   不同 Agent 负责规划、生成、校验和修复。

5. Closed-loop Self-repair
   通过日志、截图、历史尝试持续修复。

6. Structured Intermediate Representation
   用结构化 JSON 连接各阶段，降低失真。
```

最后一句：

> 它体现的是从 prompt engineering 到 agent engineering 的转变。

# 26. 如果面试官让你批判这个系统，你怎么说？

**回答话术：**

我会从几个方面批判：

```text
1. 多 Agent 增加系统复杂度
   需要证明拆分带来的收益大于成本。

2. skills 选择如果不准，会造成负迁移
   加载错误 skill 可能比不加载更差。

3. 重试机制可能消耗成本
   需要 error-aware retry 和终止策略。

4. 视觉校验不够可靠
   需要结合结构化检查。

5. Prompt 约束不够强
   最好引入 schema-constrained decoding 或 function calling。

6. 历史记忆可能污染
   需要记忆生命周期和作用域隔离。
```

结尾补一句：

> 这些问题不是否定系统，而是 Agent 工程化必须解决的问题。

# 27. 如果被问“这个系统最核心的算法点是什么？”

**回答话术：**

不要只说“大模型调用”。可以说：

> 核心不是某一个模型算法，而是 Agent 应用算法设计，包括任务分解、动态路由、结构化中间表示、工具调用、反馈重试和记忆管理。

具体可以拆成：

```text
1. Task Routing：判断单零件 / 装配体 / 需要哪些 skills
2. Planning：生成可执行结构化计划
3. Tool Use：把模型输出转成真实工具调用
4. Feedback Repair：根据执行结果进行局部修复
5. Memory Management：保留对修复有价值的信息
6. Evaluation：用执行成功率和重试次数衡量效果
```

---

# 28. 如果问“skills 动态加载是不是就是 RAG？”

**回答话术：**

有相似点，但不完全一样。

相似点：

```text
都需要检索相关知识
都把外部内容注入上下文
```

不同点：

```text
RAG 多数是为了回答问题
skills 是为了指导行动和代码生成
skills 通常有输入输出、步骤、模板和适用条件
skills 还会和工具执行反馈绑定
```

一句话：

> skills 更像 action-oriented RAG，是面向执行任务的动态能力加载。

---

# 29. 如果问“怎么减少成本？”

**回答话术：**

可以从几层优化：

```text
1. 小模型做路由，大模型做复杂生成
2. skill metadata 先召回，避免加载全文
3. 缓存相似任务的 plan 和代码模板
4. 失败日志先程序分类，再决定是否调用大模型
5. 对简单任务走模板，不走完整 Agent 链路
6. 限制最大重试次数
7. 对每一步做 early exit
```

包装版：

> 成本优化的核心是不要让大模型参与所有事情，而是把确定性逻辑、模板和缓存尽量前置。

---

# 30. 如果问“如何判断应该追问用户还是自动修复？”

**回答话术：**

可以按信息缺失类型判断：

```text
如果缺的是用户意图：追问
如果缺的是工程默认参数：自动假设
如果缺的是代码/API错误：自动修复
如果缺的是安全或关键约束：追问
```

例如：

```text
孔径没给，但可以默认：自动假设
零件到底是齿轮还是法兰不清楚：追问
SolidWorks 保存路径错误：自动修复
装配关系矛盾：追问或回到规划
```

一句话：

> 判断标准是：这个问题是否会改变用户设计意图。会改变就追问，不会改变就自动修复。

---

## 最后一段高阶总结

面试最后可以这样收尾：

> 我理解这个项目不是单纯的 CAD 代码生成，而是一个受控的 Agentic Workflow。它通过结构化任务表示、多 Agent 分工、skills 动态加载、工具执行和反馈重试，把不稳定的大模型能力包进一个可验证、可恢复的工程闭环里。真正难点不在“调用模型”，而在如何让 Agent 有边界地使用工具、管理记忆、选择技能、控制重试，并用可量化指标评估整个闭环是否真的变好。

# 多智能体 CAD Agent 流程说明

## 1. 项目目标

本项目面向 **SolidWorks 自动化建模与装配** 场景，构建了一套可分流、可校验、可重试的多智能体系统。  
系统目标不是让单个大模型一次性生成全部 CAD 代码，而是将任务拆成多个职责明确的 Agent，分别负责：

- 需求澄清与任务分流
- 装配规划
- 单零件建模
- 静态一致性校验
- 装配代码生成与执行

这样可以让系统在面对复杂装配任务时，更稳定地完成“需求理解 -> 结构规划 -> 分件建模 -> 装配落地”的全过程。

---

## 2. 系统总体思路

系统入口位于 [main.py](d:/a_src/python/sw_agent/multi_agent_src/main.py)，核心编排逻辑位于 [orchestrator.py](d:/a_src/python/sw_agent/multi_agent_src/orchestrator.py)。

整体采用“**先识别需求类型，再按任务形态分流**”的设计：

- 如果用户需求是 **单零件需求**，则直接进入零件建模流程
- 如果用户需求是 **装配需求**，则先进行装配规划，再拆分为多个零件建模任务，最后进入装配流程

这使系统既能处理简单零件，也能扩展到更大规模的多零件装配场景。

---

## 3. 总体流程图

```text
用户输入
  ↓
需求分析 Agent
  ↓
识别 request_type
  ├─ part
  │    ↓
  │  单零件工作区创建
  │    ↓
  │  零件建模 Agent
  │    ↓
  │  代码执行与动态检查
  │    ↓
  │  输出 .SLDPRT / 日志 / JSON
  │
  └─ assembly
       ↓
     装配工作区创建
       ↓
     装配规划 Agent
       ↓
     生成完整装配 JSON
       ↓
     拆分每个零件 JSON
       ↓
     多个零件建模 Agent
       ↓
     代码执行
       ↓
     零件静态校验 Agent
       ↓
     装配 Agent
       ↓
     装配代码执行与检查
       ↓
     输出 .SLDASM / 日志 / JSON
```

## 4. 各 Agent 分工

## 4.1 需求分析 Agent

对应类：`RequirementAgent`

职责：

- 与用户多轮对话，澄清需求
- 判断任务属于 `part` 还是 `assembly`
- 在信息足够后输出 `[CONFIRMED] + JSON`

输出的关键信息包括：

- `request_type`
- `name`
- `summary`
- `key_requirements`
- `assumptions`
- `requested_outputs`
- `interfaces`

这一层的作用是把自然语言需求，收敛为后续系统可直接消费的结构化输入。

---

## 4.2 装配规划 Agent

对应类：`AssemblyPlanningAgent`

仅在 `assembly` 路径下启用。

职责：

- 将确认后的装配需求转换为完整装配规划 JSON
- 为装配体拆分出所有必要零件
- 为每个零件规划可独立建模的信息
- 为后续装配提供稳定接口和配合约束

规划结果包含：

- 装配体名称与摘要
- 全局坐标系
- 全局设计规则
- 所有零件 `parts`
- 装配顺序 `assembly_sequence`
- 配合约束 `constraints`
- 装配输出路径 `assembly_output`

这个 Agent 的核心价值是：**把装配任务拆成“每个零件都能独立完成”的子任务**。

---

## 4.3 零件建模 Agent

对应类：`PartModelingAgent`

职责：

- 接收单个零件 JSON
- 生成可执行的 Python 建模代码
- 显式保存到指定 `model_file`
- 在代码中体现接口名称，方便装配阶段引用

无论是单零件任务，还是装配拆分出来的零件任务，最终都通过这一类 Agent 进行代码生成。

---

## 4.4 零件静态校验 Agent

对应类：`PartValidationAgent`

仅在 `assembly` 路径下启用。

职责：

- 校验零件代码是否与完整装配规划一致
- 检查接口名称、方向关系、保存路径是否覆盖
- 判断这段代码是否大概率支持后续装配继续进行

输出格式非常简洁：

```json
{"pass": true, "feedback": "简短意见"}
```

如果静态校验失败，系统会把简短反馈回传给零件建模 Agent，促使其重新生成代码，形成闭环。

---

## 4.5 装配 Agent

对应类：`AssemblyAgent`

仅在 `assembly` 路径下启用。

职责：

- 接收完整装配规划 JSON 和各零件输出路径
- 生成装配 Python 代码
- 按顺序插入零件
- 按规划执行平面配合和轴同心配合
- 保存最终 `.SLDASM`

这一层是从“多个已完成零件”走向“完整装配结果”的最后一步。

---

## 5. 两条主执行路径

## 5.1 零件需求路径

适用场景：

- 用户只要求建一个零件
- 不涉及多个零件之间的装配关系

执行过程：

1. 需求分析 Agent 判定 `request_type = part`
2. 系统创建单零件工作区
3. 生成 `part_request.json` 和该零件专属 `spec.json`
4. 零件建模 Agent 生成代码
5. 执行代码，完成动态检查
6. 成功后输出零件文件、日志和 JSON

这一条路径强调快速完成单零件自动建模。

---

## 5.2 装配需求路径

适用场景：

- 用户需求包含多个零件
- 存在面贴合、轴同心、插入顺序等关系
- 最终目标是 `.SLDASM`

执行过程：

1. 需求分析 Agent 判定 `request_type = assembly`
2. 系统创建装配工作区
3. 装配规划 Agent 生成完整装配规划 JSON
4. 系统为每个零件创建独立路径和独立 JSON
5. 零件建模 Agent 逐个生成零件代码并执行
6. 零件静态校验 Agent 检查是否满足装配要求
7. 所有零件通过后，装配 Agent 生成装配代码
8. 执行装配代码并保存 `.SLDASM`

这一条路径体现了系统“**先规划，再分发，再汇总装配**”的集群式设计思想。

## 6. 工作空间与数据组织

系统会自动在如下基目录下创建工作区：

`D:\a_src\python\sw_agent\agent_output`

### 单零件任务目录结构

```text
装配体名或零件名-时间戳/
  part/
  json/
  logs/
```

### 装配任务目录结构

```text
装配体名-时间戳/
  parts/
    part_a/
    part_b/
    ...
  assembly/
  json/
  logs/
```

每次任务都会独立创建新目录，避免不同任务间的结果相互覆盖。

## 7. JSON 在系统中的作用

本项目中，JSON 不是简单日志，而是 **Agent 之间的标准通信协议**。

它承担了三层作用：

- 需求确认结果的结构化表达
- 装配规划结果的中间表示
- 单个零件任务的可分发子任务载体

尤其在装配场景中，每个零件都会拥有独立 JSON 子对象，其中包含：

- 零件名称
- 功能
- 形状描述
- 关键尺寸
- 面接口
- 轴接口
- 点接口
- 与装配相关的关系说明
- 自己的工作区路径

这保证了每个零件 Agent 拿到自己的 JSON 后，就可以独立完成建模，而不必重新理解整套装配语义。

---

## 8. 校验与重试机制

本项目强调“**生成不是终点，执行和校验才是闭环**”。

### 零件阶段

- 建模代码生成后，先执行
- 如果执行失败，系统把错误日志返回给建模 Agent 重新生成
- 如果执行成功，装配场景下还会进入静态校验
- 若静态校验失败，系统把简短意见再反馈给建模 Agent

### 装配阶段

- 装配代码生成后先执行
- 如果执行失败，则将执行日志回传给装配 Agent 重写代码
- 如果执行成功，还会做基础静态规则检查

这一机制提升了整体系统在复杂任务下的稳健性。

---

## 9. 知识库在流程中的作用

项目为每个 Agent 配置了独立知识库目录，位于：

[knowledge_base](d:/a_src/python/sw_agent/multi_agent_src/knowledge_base)

包括：

- `requirement_agent`
- `assembly_planning_agent`
- `part_modeling_agent`
- `part_validation_agent`
- `assembly_agent`

它们分别承担不同类型的知识约束：

- 需求分析知识：能力边界、需求澄清规则、`part/assembly` 判定规则
- 装配规划知识：零件拆分原则、接口命名规则、约束规划规则
- 零件建模知识：SolidWorks 封装 API、单位换算、建模范式
- 零件校验知识：路径、接口、单位、结构一致性检查
- 装配知识：插入零件、配合约束、装配保存与日志策略

因此，这套系统不是“裸模型直出”，而是“**Agent + 专属知识库**”协同工作。

---

## 10. 系统亮点

从组会展示角度，这个项目有几个比较清晰的亮点：

- 从单 Agent 直出，升级为多 Agent 分工协同
- 能自动识别零件需求与装配需求，并动态分流
- 装配任务中会先做结构化规划，而不是直接盲目生成代码
- 每个零件拥有独立 JSON 和独立工作空间，具备天然可扩展性
- 引入了“执行校验 + 静态校验 + 失败反馈重试”的闭环机制
- 每个 Agent 都绑定专属知识库，行为更受控、输出更稳定
